# DX01 — Skill `code-documentation` (agnóstica a linguagem e idioma)

**Contexto:** projetos que usam o framework frequentemente precisam de diretrizes internas sobre comentários e documentação inline. Sem skill dedicada, cada projeto reinventa o padrão: alguns exigem JSDoc em tudo, outros só headers, outros nada. Resultado: divergência de estilo entre sub-projetos, onboarding mais lento, revisão de PR gastando tempo com "precisa documentar?" em vez de conteúdo. O framework já cobre testes (skill `testing`), commits (`GIT_CONVENTIONS.md`), segurança (`security-review`), mas não documentação inline. Esta skill fecha o gap de forma agnóstica: princípios universais fixos + configuração por projeto + exemplos por linguagem.

**Abordagem:** skill distribuída `code-documentation` em três camadas:

### 1. `SKILL.md` — princípios universais (não configuráveis)

O corpo da skill contém:

- **Regra central:** comentar **por quê**, não **o quê**. Código já mostra a ação.
- **Quando é obrigatório** (multi-nível, por default; ajustável via tabela de configuração):

| Nível | Alvo | Política default |
|---|---|---|
| Arquivo | header no topo do módulo | Obrigatório em paths declarados |
| Classe / tipo / interface | classe pública, tipo/interface exportado | Obrigatório quando exportado |
| Função / método | função/método público | Obrigatório quando exportado; recomendado em internos complexos |
| Variável / constante | constante pública, config exportada | Obrigatório quando exportada |
| Enum / union type | estados, categorias | Obrigatório quando exportado |
| Bloco de código | workaround, regex complexa, algoritmo não-óbvio | Comentário inline obrigatório |
| TODO / FIXME | débito técnico marcado | Obrigatório com contexto (por quê + quando remover + ID do backlog se houver) |

- **O que cada doc deve responder:**
  - Arquivo/classe: papel no sistema; fronteira com outros módulos
  - Função: contrato (parâmetros, retorno, exceções), decisões não-óbvias, exemplo de uso quando não trivial
  - Variável/const: razão de existir, impacto se mudar, relação com outras partes
  - Bloco: por que este caminho específico (não o óbvio)

- **Anti-padrões** (nunca fazer):
  - Comentário que repete o identificador (`// set user name` antes de `setUserName`)
  - Doc desatualizado (comentário diz uma coisa, código faz outra)
  - `TODO` sem contexto (nome, data ou ID de backlog) e sem prazo de retirada

- **Triggers** (quando a skill deve ser acionada): criar arquivo novo em área declarada; adicionar função pública sem doc; modificar lógica complexa sem comentário; PR review detecta ausência.

### 2. Tabela `## Configuração do projeto` — preenchida no setup, editável depois

A skill distribuída vem com esta tabela no final, com valores `{Adaptar:...}` preenchidos no setup com base em detecção:

```markdown
## Configuração do projeto

| Config | Valor | Notas |
|---|---|---|
| Idioma dos comentários | {Adaptar: pt-BR / en / es / ...} | Editar se preferir outro idioma |
| Formato principal | {Adaptar: JSDoc / TSDoc / docstring-google / ...} | Formato canônico da linguagem principal |
| Formato secundário | {Adaptar: — / ...} | Preencher em projetos multi-linguagem |
| Áreas que exigem header de arquivo | {Adaptar: ex: backend/services/, backend/routes/, frontend/hooks/} | Lista de paths; fora delas, header é opcional |
| Nível obrigatório mínimo | {Adaptar: exports-publicos / todos-exports / tudo-publico-e-interno-complexo} | Quão estrito o projeto é |
| Documentar variáveis públicas | {Adaptar: true / false} | |
| Documentar tipos/interfaces | {Adaptar: true / false} | |
| Comentário inline em blocos complexos | true | Fixo; padrão universal |
| TODO/FIXME exige contexto | true | Fixo; padrão universal |
```

- Tabela é **parte da skill distribuída**, não arquivo externo.
- Setup preenche os `{Adaptar:}` com inferência (ver passo 3). Dev edita a tabela manualmente quando quiser mudar.
- Update **preserva** a tabela se já preenchida (é customização local).

### 3. `skills/code-documentation/EXAMPLES/<lang>.md` — biblioteca no framework

Diretório com um arquivo por linguagem/formato suportado. Arquivos iniciais:

- `EXAMPLES/jsdoc.md` — JavaScript
- `EXAMPLES/tsdoc.md` — TypeScript
- `EXAMPLES/python-google.md` — Python (Google style)
- `EXAMPLES/python-numpy.md` — Python (NumPy/SciPy style)
- `EXAMPLES/rustdoc.md` — Rust
- `EXAMPLES/godoc.md` — Go
- `EXAMPLES/javadoc.md` — Java
- `EXAMPLES/phpdoc.md` — PHP

Cada arquivo tem 3-5 exemplos concretos cobrindo: header de arquivo, classe/tipo, função, variável/const, bloco inline, TODO com contexto. Todos os exemplos são **neutros de idioma** (texto em `{Idioma}` como placeholder) — o setup substitui pelo idioma detectado ao injetar na skill.

Ao criar uma linguagem nova no futuro, basta adicionar um arquivo em `EXAMPLES/` — SKILL.md não muda.

### 4. Padrão de injeção no setup (híbrido, opção C)

Setup detecta stack(s) via `map-codebase` / scan de arquivos:

1. **Linguagem principal** (inferência): maior volume em `backend/`, `src/`, `lib/`. Resolver para arquivo em `EXAMPLES/` (ex.: JS → `jsdoc.md`).
2. **Linguagem secundária** (se houver): detectada em `frontend/`, `e2e/`, `scripts/`.
3. **Idioma**: detectado via amostragem de comentários/docs existentes no projeto. Se inconclusivo ou projeto greenfield, perguntar.
4. **Áreas que exigem header**: inferidas pela estrutura existente (paths onde já há documentação ou onde convenção pede). Em greenfield, sugerir defaults da linguagem (ex.: JS → `src/services/`, `src/routes/`) e perguntar.
5. Copiar `SKILL.md` da skill para o projeto.
6. **Injetar** na seção `## Exemplos` da skill gerada no projeto:
   - Conteúdo de `EXAMPLES/{linguagem-principal}.md`
   - Conteúdo de `EXAMPLES/{linguagem-secundária}.md` se detectada
   - Substituir `{Idioma}` pelo idioma configurado
7. Preencher a tabela `## Configuração do projeto` com os valores inferidos.

Update segue o mesmo fluxo para projetos que não têm a skill; projetos que já têm mantêm configuração e apenas recebem novas áreas do SKILL.md via structural merge.

### 5. Monorepo

Em monorepo (MR1), a skill gerada no L0 CLAUDE.md pode referenciar configurações diferentes por sub-projeto. Dois sub-projetos com stacks distintas (ex.: backend Node + frontend TypeScript) recebem dois blocos de exemplos e uma coluna adicional na tabela indicando escopo (`backend/**`, `frontend/**`).

**Alternativas descartadas:**
- **(A) Exemplo único inline no SKILL.md** — simples, mas não escala para multi-stack e obriga editar SKILL.md por linguagem nova.
- **(B) Biblioteca EXAMPLES/ referenciada sem injeção** — skill gerada aponta para `@EXAMPLES/<lang>.md` do framework; projeto precisa manter acesso ao framework source. Perde auto-contenção.
- **Skill com idioma fixo (pt-BR ou en)** — viola princípio de projetos escolherem seu idioma; força retrabalho para projetos em outros idiomas.
- **Prescrever formato único (ex.: só JSDoc)** — inviável em projetos multi-stack ou fora do ecossistema JS.

**Critérios de aceitação:**
- [ ] `skills/code-documentation/SKILL.md` criado com: frontmatter (`name`, `description`, `user_invocable`), seções obrigatórias (Quando usar, Quando NÃO usar, Checklist, Regras), regras multi-nível, lista de anti-padrões, triggers explícitos, tabela `## Configuração do projeto` com placeholders `{Adaptar:}`, seção `## Exemplos` placeholder para injeção
- [ ] `skills/code-documentation/EXAMPLES/` com pelo menos: jsdoc.md, tsdoc.md, python-google.md, rustdoc.md, godoc.md (5 linguagens no release inicial)
- [ ] Cada arquivo em EXAMPLES/ tem exemplos para os 6 níveis (arquivo, classe/tipo, função, variável, bloco, TODO) e usa `{Idioma}` como placeholder textual
- [ ] Skill distribuída via `setup-framework`: cópia do SKILL.md + injeção da seção Exemplos com conteúdo do(s) arquivo(s) de EXAMPLES/ relevante(s) + preenchimento da tabela Configuração
- [ ] Setup detecta linguagem e idioma (com fallback para pergunta ao dev quando inconclusivo)
- [ ] `update-framework` adiciona a skill a projetos que não têm, preservando configuração quando já existe (structural merge na tabela)
- [ ] MANIFEST.md registra `skills/code-documentation/` (tier apropriado) e `skills/code-documentation/EXAMPLES/` (não copiado diretamente — consumido pelo setup)
- [ ] `CLAUDE.md` template (distribuído) lista a skill na tabela de skills
- [ ] `docs/SKILLS_MAP.md` inclui a skill na seção "Skills de suporte" com dependências
- [ ] Sincronia source↔template passa no `check-sync.sh`
- [ ] Monorepo: skill suporta múltiplos blocos de exemplos e escopo por sub-projeto

**Restrições:**
- Skill **não** prescreve idioma, linguagem ou formato — tudo configurável
- Skill **não** prescreve cobertura 100% (cada projeto escolhe o nível via "Nível obrigatório mínimo")
- EXAMPLES/ é consumido pelo setup, **não** distribuído para projetos (projetos recebem o conteúdo injetado na skill, não o diretório)
- Não reinventar documentação que já está em ferramentas da linguagem (ex.: não fazer typedoc — apenas ensinar a escrever o doc que o typedoc vai processar)
- Preservar customização local em update: tabela de configuração é read-only para o merge structural
