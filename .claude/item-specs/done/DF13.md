# DF13 — Discovery Routing + `/quick`

**Contexto:** o fluxo anterior forçava `/spec` para qualquer mudança, inclusive correções triviais (typo, bump de dependência, ajuste de config, rename sem lógica). Isso gerava overhead de cerimônia que não agregava rastreabilidade — a spec virava checkbox vazio. Pesquisa do ecossistema (OpenSpec, cc-sdd) mostrou que frameworks maduros fazem triagem antes de escolher o fluxo.

**Abordagem:** gate de triagem anterior ao `spec-driven` classifica o trabalho em três faixas:
1. **Quick task** — correção trivial, sem lógica nova → `/quick` (sem spec, sem STATE.md, sem DoD completo)
2. **Spec única** — item cabe em uma spec → fluxo normal
3. **Multi-spec** — escopo grande, precisa decomposição → `/prd`

A skill `/quick` (`skills/quick/SKILL.md`) implementa o fast-path com critérios explícitos: "toca lógica de negócio? toca auth/pagamentos? cria abstração nova? afeta >3 arquivos? altera schema?". Se qualquer resposta é sim, redireciona para `/spec`. DoD simplificado: verify.sh + Conventional Commits + PR.

Alternativas descartadas:
- Tornar `/quick` uma flag do `/spec` — dilui a triagem; dev ignora a flag e sempre cria spec.
- Deixar a decisão livre para o dev — mantém o overhead atual, não resolve o problema.
- Exigir spec mínima mesmo em quick task — cerimônia sem valor, a rastreabilidade já vem do commit + PR.

**Critérios de aceitação:**
- [x] Slash command `/quick` em `skills/quick/SKILL.md` com frontmatter `user_invocable: true`
- [x] Critérios explícitos "quando usar" e "quando NÃO usar" (lista fechada)
- [x] Redirecionamento automático para `/spec` quando critérios falham
- [x] Fluxo simplificado: implementar → testar → verify.sh → commit → PR
- [x] DoD simplificado para quick task documentado
- [x] Integração no `CLAUDE.template.md` (skill listada como primeira opção de fluxo)
- [x] Distribuída via `setup-framework` (`templates/.claude/skills/quick/SKILL.md`)
- [x] Listada em `docs/SKILLS_MAP.md` (seção "Skills de suporte")
- [x] MANIFEST.md atualizado

**Restrições:**
- `/quick` **não** serve para lógica de negócio — mudança em regra, fluxo ou comportamento exige spec.
- `verify.sh` continua obrigatório — quick não pula verificação.
- Se testes quebrarem durante `/quick`, parar imediatamente e migrar para `/spec` (sinal de que a mudança não era trivial).
- Não expandir a lista de critérios sem discussão — quanto mais critérios, mais frágil a triagem.
