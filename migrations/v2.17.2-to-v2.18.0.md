# Migration v2.17.2 → v2.18.0

> Data: 2026-04-08
> Tipo: minor

## Resumo

Worktrees, entrega via PR e decomposicao obrigatoria agora sao regras ativas (antes eram comentarios HTML ou opcionais). Setup passa a configurar .gitignore para artefatos do framework. Marketplace.json adicionado para distribuicao de plugin.

## Pre-requisitos

- Versao atual do projeto: v2.17.2
- Se estiver em versao anterior, aplique as migrations anteriores primeiro

## Mudancas

### 🔄 Overwrite (substituir arquivo inteiro)

#### 1. `.claude-plugin/plugin.json`

**O que mudou:** Campo `version` atualizado para 2.18.0.

**Como aplicar:** Copiar do framework.

```bash
cp {framework_path}/.claude-plugin/plugin.json {project_path}/.claude-plugin/plugin.json
```

**Impacto se nao aplicar:** Versao reportada pelo plugin fica desatualizada.

---

### 📝 Structural (adicionar/remover secoes)

#### 2. `docs/GIT_CONVENTIONS.md`

**Secoes adicionadas:**

Adicionar apos a secao `## Pull Requests` e antes de `**Título:**`:

```markdown
### Regra de entrega

Toda entrega e via Pull Request. Push direto para `main` (ou branch de integracao) e proibido. Esta regra se aplica a sessoes humanas e a sessoes de IA.
```

**Impacto se nao aplicar:** Regra de entrega via PR nao fica documentada em GIT_CONVENTIONS.

---

### 👀 Manual (decisao humana)

#### 3. `CLAUDE.md` (gerado pelo setup)

**O que mudou no template — 3 secoes:**

**3a. Secao "Worktrees e subagents"** — conteudo antes comentado agora esta ativo:

```markdown
### Worktree por sessao (recomendado)

Cada sessao de trabalho deve rodar numa worktree isolada para nao interferir no working directory principal.
Worktrees ficam em `.claude/worktrees/` (ja no .gitignore).

Ao iniciar sessao em feature/fix:
- Criar ou reutilizar worktree para a branch (ex: `feat/xyz` → `.claude/worktrees/feat-xyz`)
- Todo o trabalho da sessao acontece nessa worktree
- Ao finalizar (PR aberto) → limpar worktree

### Subagents e isolamento

| Tipo de subagent | Onde roda | Motivo |
|---|---|---|
| **Read-only** (auditoria, validacao, report, explore) | Na worktree da sessao (sem worktree nova) | Ve o codigo em progresso, nao o main. Rapido, sem overhead. |
| **Write exploratorio** (refactor, spike, prototipagem) | Worktree propria (nova) | Isola mudancas experimentais. Se boas, merge traz de volta. |
```

**3b. Secao "Paralelismo dentro de um item"** renomeada para "Decomposicao e paralelismo dentro de um item (obrigatorio para medio+)" com fluxo explicito de decomposicao.

**3c. Nova secao "Entrega via Pull Request (obrigatorio)"** adicionada apos "Antes de commitar":

```markdown
## Entrega via Pull Request (obrigatorio)

Sessoes de trabalho NUNCA fazem push direto para `main` (ou branch principal). Toda entrega e via Pull Request.

### Regras absolutas

- **NUNCA `git push` direto para `main`.** Sempre via PR.
- **NUNCA `git push --force` para `main`.** Sem excecoes.
- Se o projeto usa branch `develop`, mesma regra: nunca push direto, sempre PR.
```

**Sugestao:** Aplicar as 3 secoes. Se o projeto nao usa worktrees, a secao 3a pode ser removida (tem `{Adaptar:}` para isso). As secoes 3b e 3c sao fortemente recomendadas.
