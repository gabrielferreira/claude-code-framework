# Migration v2.36.0 → v2.37.0

> Data: 2026-04-10
> Tipo: minor

## Resumo

Nova skill `/discuss` — fase conversacional estruturada antes da spec. Faz scout no codebase, identifica gray areas, guia deep-dive e gera spec completa ao final com decisoes incorporadas. Dual-mode (repo + Notion).

## Pre-requisitos

- Versao atual do projeto: v2.36.0
- Se estiver em versao anterior, aplique as migrations anteriores primeiro

## Mudancas

### 🆕 Arquivos novos

#### 1. `.claude/skills/discuss/SKILL.md` (novo)

**O que e:** Skill `/discuss` — passo anterior ao `/spec` para features com gray areas, dominio novo ou escopo vago. Faz scout no codebase, identifica ambiguidades automaticamente, guia deep-dive em cada uma e gera spec ao final.

**Relevante se:** o projeto tem features com escopo incerto ou decisoes tecnicas nao triviais antes de especificar.

**Como instalar:** Copiar `skills/discuss/SKILL.md` para `.claude/skills/discuss/SKILL.md`.

```bash
mkdir -p .claude/skills/discuss
cp {framework_path}/skills/discuss/SKILL.md .claude/skills/discuss/SKILL.md
```

**Se decidir nao instalar:** o fluxo continua funcionando — `/spec` pode ser usado diretamente. A skill e recomendada, nao obrigatoria.

### 📝 Structural — secoes novas/removidas

#### 2. `docs/SKILLS_GUIDE.md`

**Secoes adicionadas:**

Adicionar apos a secao `## spec-creator (/spec)` (ou antes de `## prd-creator (/prd)`):

```markdown
## discuss (`/discuss`)

**O que é:** Fase conversacional estruturada antes da spec. Faz scout no codebase, identifica gray areas (ambiguidades, decisões técnicas, dependências, conflitos), guia deep-dive em cada uma e gera spec completa ao final com decisões incorporadas. Substitui o salto direto de "quero X" para `/spec` quando há incerteza.

**Quando usar:**
- Feature com 2+ interpretações possíveis da solução técnica
- Decisões técnicas não triviais (escolha de lib, pattern, schema)
- Domínio novo (primeira feature nessa área do projeto)
- Escopo vago onde critérios de aceitação ainda não estão claros

**Output:** Spec gerada (repo ou Notion) com decisões consolidadas, gray areas exploradas documentadas como "Notas — Decisões do /discuss", e gray areas não exploradas marcadas como `{A DETALHAR}` com impacto documentado.
```

#### 3. `docs/SKILLS_MAP.md`

**Secoes adicionadas:**

Na tabela de skills, adicionar linha antes de `spec-creator`:

```markdown
| discuss (`/discuss`) | Ao explorar gray areas, dominio novo ou escopo vago antes de especificar — scout + decisoes + spec gerada ao final |
```

#### 4. `docs/WORKFLOW_DIAGRAM.md`

**Secoes adicionadas:**

No diagrama ASCII, entre o gate "Nao-trivial?" e "/spec", adicionar o gate de `/discuss`. O diagrama atualizado inclui um novo decision box "Gray areas? Dominio novo? Escopo vago?" que bifurca entre `/discuss` (gera spec ao final) e `/spec` direto.

**Impacto se nao aplicar:** diagrama fica sem o passo `/discuss`, mas nao quebra nada.

### 🔧 Content patches — mudancas dentro de secoes existentes

#### 5. `docs/QUICK_START.md` — secao `### 1. Especificar`

**Motivo:** adicionado tip sobre `/discuss` como alternativa antes do `/spec`.

**Texto antigo:**
```markdown
Descreva a feature ou tarefa. O comando gera uma spec estruturada em `.claude/specs/` com criterios de aceite, escopo e definicao de done.

### 2. Implementar
```

**Texto novo:**
```markdown
Descreva a feature ou tarefa. O comando gera uma spec estruturada em `.claude/specs/` com criterios de aceite, escopo e definicao de done.

> **Tip:** Se a feature tem gray areas, dominio novo ou escopo vago, use `/discuss` antes. Ele faz scout no codebase, guia decisoes sobre ambiguidades e gera a spec ao final com as decisoes incorporadas.

### 2. Implementar
```

**Impacto se nao aplicar:** usuario nao descobre o `/discuss` pelo quick start, mas funciona normalmente.

#### 6. `docs/SPEC_DRIVEN_GUIDE.md` — secao de sessoes separadas (Research/Plan/Implement)

**Motivo:** adicionada etapa "Discuss" entre Research e Plan no fluxo RPI.

**Texto antigo:**
```markdown
1. **Research:** seguir protocolo da skill research (`.claude/skills/research/README.md`). Investigar código existente, patterns de reuso, dependências, riscos. Output: `.claude/specs/{id}-research.md` com achados estruturados por 6 eixos.
2. **Plan:** escrever spec, design doc, breakdown de tasks a partir dos achados do research. Output: spec aprovada + tasks priorizadas.
3. **Implement:** executar tasks em waves (sequenciais → paralelas → integração). Output: código + testes.
```

**Texto novo:**
```markdown
1. **Research:** seguir protocolo da skill research (`.claude/skills/research/README.md`). Investigar código existente, patterns de reuso, dependências, riscos. Output: `.claude/specs/{id}-research.md` com achados estruturados por 6 eixos.
2. **Discuss (se gray areas):** usar `/discuss` para resolver ambiguidades, decisões técnicas e dependências não resolvidas. Faz scout no codebase, guia deep-dive em cada gray area e gera a spec ao final com decisões incorporadas. Se não há gray areas, pular para Plan com `/spec` direto.
3. **Plan:** escrever spec (se não gerada pelo `/discuss`), design doc, breakdown de tasks a partir dos achados do research. Output: spec aprovada + tasks priorizadas.
4. **Implement:** executar tasks em waves (sequenciais → paralelas → integração). Output: código + testes.
```

**Impacto se nao aplicar:** guia nao menciona o passo Discuss, mas o fluxo continua valido.
