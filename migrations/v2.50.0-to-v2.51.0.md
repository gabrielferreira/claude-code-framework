# Migration v2.50.0 → v2.51.0

> Data: 2026-05-13
> Tipo: minor

## Resumo

A escala de Estimativa deixa de ser hard-coded no framework e passa a ser definida por cada projeto em `.claude/conventions/estimation.md`. O mapeamento direto Complexidade → Estimativa foi removido — os dois eixos sao independentes. Skills `/backlog-update`, `/spec` e `spec-creator` agora leem a escala desse arquivo.

## Pre-requisitos

- Versao atual do projeto: v2.50.0
- Se estiver em versao anterior, aplique as migrations anteriores primeiro

## Mudancas

### 🆕 Arquivos novos

#### 1. `.claude/conventions/estimation.md`

**O que e:** novo arquivo de convencao do projeto que define a escala de Estimativa usada no backlog e nas specs. Substitui a escala fixa `15min | 30min | 1h | 2h | 4h | 1d | 2d | 1sem` que estava hard-coded nas skills.

**Por que:** times diferentes querem escalas diferentes (Fibonacci, T-shirt, horas, story points). Estimativa nao e decisao do framework, e do projeto. Alem disso, Estimativa e Complexidade agora sao eixos independentes — Complexidade determina cerimonia (spec light, execution-plan), Estimativa e tempo-de-pessoa real (ou outro criterio do time).

**Como aplicar:**

Rodar `/update-framework` — a auditoria de completude detecta o arquivo ausente (severidade 🟠 alto, pois `/backlog-update`, `/spec` e `spec-creator` precisam dele) e oferece criar via wizard com 4 presets:

- **Horas/dias** (escala anterior do framework): `15min`, `30min`, `1h`, `2h`, `4h`, `1d`, `2d`, `1sem`
- **Fibonacci**: `1`, `2`, `3`, `5`, `8`, `13`, `21`
- **T-shirt**: `PP`, `P`, `M`, `G`, `GG`
- **Customizar**: instala template vazio com tabela de exemplo; o usuario preenche depois

**Criacao manual** (sem rodar `/update-framework`):

```bash
mkdir -p .claude/conventions
cat > .claude/conventions/estimation.md <<'EOF'
<!-- framework-tag: v2.51.0 framework-file: conventions/estimation.md -->
# Estimativa (convencao do projeto)

Definir aqui a escala de Estimativa usada no projeto, em **qualquer formato** (pontos Fibonacci, horas, T-shirt P/M/G, story points, etc).

**Independente da Complexidade.** Complexidade determina cerimonia (spec light, execution-plan, research). Estimativa e um eixo separado.

## Valores validos

| Valor | Significado |
|-------|-------------|
| 15min | tarefa rapida |
| 30min | meia hora |
| 1h    | uma hora |
| 2h    | duas horas |
| 4h    | meio dia |
| 1d    | um dia |
| 2d    | dois dias |
| 1sem  | uma semana |

Na duvida, classificar para cima.
EOF
```

(Ajustar a tabela ao formato que o time decidir.)

**Impacto se nao aplicar:** `/backlog-update add`, `/spec` e `spec-creator` vao alertar e bloquear ao chegar no campo Estimativa, ate o arquivo ser criado.

**Apos criado:** o arquivo e propriedade do projeto. `/update-framework` nunca sobrescreve. Edite a qualquer momento para mudar a escala.

### 🔧 Migracoes de gitignore (todos os modos)

Versoes anteriores nao previam que `.claude/specs/{id}-plan.md` (execution-plan) e `.claude/specs/{id}-research.md` (research) sao artefatos transientes — criados em fase plan, validados em fase done, descartaveis apos cada PR. Hoje o fluxo manda deletar manualmente antes do PR. Esta migration adiciona ao `.gitignore` padroes que ignoram esses arquivos automaticamente.

**Aplicar:** `/update-framework` faz append automatico no `.gitignore` (Fase 3.7). Manualmente:

```bash
grep -qxF ".claude/specs/*-plan.md" .gitignore || echo ".claude/specs/*-plan.md" >> .gitignore
grep -qxF ".claude/specs/*-research.md" .gitignore || echo ".claude/specs/*-research.md" >> .gitignore
```

**Se ja existem arquivos trackeados** (`{id}-plan.md` ou `{id}-research.md` commitados):

```bash
# Listar
git ls-files '.claude/specs/*-plan.md' '.claude/specs/*-research.md'

# Remover do tracking (coordenar com o time — afeta working tree de outros devs)
git rm --cached '.claude/specs/*-plan.md' '.claude/specs/*-research.md'
git commit -m "chore: stop tracking transient execution-plan/research artifacts"
```

### 🔧 Migracoes de gitignore (modo Notion/externo)

**Aplica-se apenas a projetos que escolheram Notion ou ferramenta externa no setup** (detectavel via `## Integracao Notion (specs)` ou referencia a Jira/Linear/etc. no CLAUDE.md).

Em modo Notion/externo, `.claude/specs/` so recebe arquivos transientes — o setup nao instala `backlog.md`, `TEMPLATE.md`, etc. (vivem na ferramenta externa). Faz sentido gitignorar a pasta inteira, preservando `done/` caso o time queira historico local de specs concluidas.

**Aplicar:** `/update-framework` faz append automatico se detectar modo Notion/externo. Manualmente:

```bash
cat >> .gitignore <<'EOF'
# Modo Notion/externo: backlog e templates vivem fora do repo
.claude/specs/*.md
!.claude/specs/done/
EOF
```

**Se ja existem arquivos trackeados** em `.claude/specs/` (deveriam ter sido pulados pelo setup, mas pode ter sido import manual):

```bash
# Listar
git ls-files '.claude/specs/*.md' | grep -v '^\.claude/specs/done/'

# Remover do tracking (coordenar com o time)
git ls-files '.claude/specs/*.md' | grep -v '^\.claude/specs/done/' | xargs git rm --cached
git commit -m "chore: stop tracking .claude/specs/ in Notion mode"
```

### 📝 Observacoes

- **Valores antigos no backlog existente:** itens ja registrados no backlog com valores como `30min`, `2h`, `1sem` continuam validos se a escala escolhida foi "Horas/dias". Se o time escolheu Fibonacci ou T-shirt, considerar uma passada para renormalizar valores — nao e obrigatorio, pode ser feito progressivamente conforme cada item e tocado.
- **Severidade nao mudou:** o mapeamento Complexidade → Severidade (`Pequeno → baixa | Médio → media | Grande → alta | Complexo → critica`) continua vivo em `spec-creator/SKILL.md`. So Estimativa foi desacoplada.
- **Threshold de bug urgente (`<30min`)** em CLAUDE.md/spec-driven continua valido — e gate operacional para TDD, nao parte da escala de Estimativa.
