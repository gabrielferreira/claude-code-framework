# Migration v2.34.0 → v2.35.0

> Data: 2026-04-10
> Tipo: minor

## Resumo

Simplificação do framework: 3 skills e 1 agent eliminados por redundância, contradições corrigidas (agents read-only, TDD, coverage, default de execução), fast-path (DF13) para quick tasks, DoD e CLAUDE.template compactados, tríade de orquestração consolidada.

## Pre-requisitos

- Versao atual do projeto: v2.34.0
- Se estiver em versao anterior, aplique as migrations anteriores primeiro

## Mudancas

### 🗑️ Arquivos removidos do framework

#### 1. `.claude/skills/syntax-check/README.md`

**Motivo:** 100% dos checks já existem em `code-quality`. Sem perda de funcionalidade.

**Como aplicar:**
```bash
rm -rf .claude/skills/syntax-check/
```
Remover a linha correspondente na tabela "Skills" do CLAUDE.md (trigger "Vai commitar código?").

#### 2. `.claude/skills/performance-profiling/README.md`

**Motivo:** ~90% overlap com agent `performance-audit`. Usar o agent para investigações de performance.

**Como aplicar:**


#### 3. `.claude/agents/plan-checker.md`

### 📝 Structural — seções novas

#### 4. `.claude/skills/spec-driven/README.md`

**Seção nova: "Triagem: classificar antes de iniciar"** — gate de triagem (DF13 Discovery Routing) antes do fluxo. Classifica incoming em: quick task (fast-path sem spec), spec única (fluxo normal), ou multi-spec (→ /prd).

**Como aplicar:** Adicionar antes da seção "## Fluxo: da demanda ao código":

```markdown
## Triagem: classificar antes de iniciar

> **Fast-path (quick task):** Correções triviais (typo, bump de dependência, ajuste de mensagem/config, rename, fix de 1-2 linhas sem nova lógica de negócio) não precisam de spec. Implementar → testar → verify.sh → commit → PR. Sem spec, sem STATE.md, sem DoD completo. Backlog pós-facto se relevante. **Se a mudança toca lógica de negócio, não é trivial.**

Para tudo que não é quick task, seguir o fluxo abaixo:
```

#### 5. `.claude/skills/definition-of-done/README.md`

**Seção nova: "Quick tasks (fast-path)"** — DoD simplificado para quick tasks.

**Como aplicar:** Adicionar antes de "## Checklist universal":

```markdown
## Quick tasks (fast-path)

Para quick tasks (typo, bump de dependência, ajuste de mensagem/config, fix trivial sem lógica de negócio nova), o DoD é simplificado:

- [ ] `bash scripts/verify.sh` passa sem erros
- [ ] Commit segue Conventional Commits
- [ ] PR aberto (não push direto para main)

Sem spec, sem STATE.md, sem checklist completo abaixo. Se a mudança toca lógica de negócio, não é quick task — usar o checklist universal.
```

### 📝 Structural — content patches (mudanças intra-seção)

#### 7. CLAUDE.md — Regra 1 "Spec-driven obrigatório"

**Antes:** "Se não existe spec → PARAR e criar spec antes de qualquer código. Toda mudança tem spec."

**Depois:** "Classificar o trabalho: Quick task → implementar direto sem spec. Spec única → criar se não existe. Iniciativa maior → /prd antes."

#### 8. CLAUDE.md — Regra de agents (seção "Agents")

**Antes:** "Agents são para auditoria e report — NÃO para implementação direta."

**Depois:** "Agents de auditoria (read-only) devolvem relatórios — nunca fix direto. Agents de execução (task-runner, refactor-agent) são infraestrutura de orquestração em worktree isolada."

#### 9. CLAUDE.md — TDD `{Adaptar}`

**Antes:** "{Adaptar: se o projeto não usa TDD estrito, ajustar para 'testes obrigatórios'}"

**Depois:** "{Adaptar: TDD estrito é o default. Se o projeto já define política diferente, respeitar. Caso contrário manter TDD obrigatório.}"

#### 10. CLAUDE.md — Coverage

**Antes:** `Backend | 100% | ≥95%` e `Frontend | 100% | ≥90%`

**Depois:** `Backend | {X}% | {Y}%` e `Frontend | {X}% | {Y}%` com `{Adaptar}` por módulo.

#### 11. CLAUDE.md — Modo de execução

**Antes:** Default era delegar para sub-agents, exceção era implementar direto.

**Depois:** Default é implementar direto, opção é delegar se sub-agents disponíveis.

#### 12. CLAUDE.md — Tabela de agents

**Coluna "Modelo" removida.** Modelo vive no frontmatter de cada agent.

#### 13. CLAUDE.md — Seção "Modelos para sub-agents"

**Sub-tabela "Agents custom deste projeto" removida** (duplicava tabela principal). Manter hierarquia de decisão + tabela de decisão + agents built-in.

#### 14. CLAUDE.md — Worktrees

**Seção compactada.** Detalhes (tabela de tipos, regras por tipo) substituídos por regra resumida de 2 linhas.

### 👀 Manual — STATE.md

#### 15. `.claude/specs/STATE.md`

**O que mudou:** Estrutura simplificada de 7 seções para 3.

**Antes (7 seções):** Decisões arquiteturais, Blockers ativos, Execução ativa (com log de transições), Lições aprendidas, Ideias adiadas, TODOs entre sessões, Regras de manutenção.

**Depois (3 seções):** Em andamento (spec + fase + o que falta), Próximos passos, Notas (blockers + ideias adiadas).

**Como aplicar:** Revisar o STATE.md do projeto. Se tem dados ativos, migrar para a nova estrutura:
- "Execução ativa" → "Em andamento"
- "TODOs entre sessões" → "Próximos passos"
- "Blockers ativos" + "Ideias adiadas" → "Notas"
- "Decisões arquiteturais" → mover para spec ou design doc relevante
- "Log de transições" e "Lições aprendidas" → descartar (overhead sem valor)

Se STATE.md está vazio/placeholder, substituir pelo novo template.
