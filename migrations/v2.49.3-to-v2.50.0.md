# Migration v2.49.3 → v2.50.0

> Data: 2026-05-13
> Tipo: minor

## Resumo

`.claude/specs/STATE.md` agora é **pessoal por dev** — gitignored por padrão. Versões anteriores commitavam STATE.md no repo do projeto, gerando conflitos de merge garantidos e contaminação de contexto entre devs em projetos multi-dev. Esta versão adiciona a entrada no `.gitignore` distribuído e amplia a regra #6 do `CLAUDE.md` template para explicitar o princípio (vale também para `.claude/plans/`, que já era gitignored).

## Pre-requisitos

- Versao atual do projeto: v2.49.3
- Se estiver em versao anterior, aplique as migrations anteriores primeiro

## Mudancas

### 🔄 Overwrite (substituir arquivo inteiro)

#### 1. `.claude/skills/update-framework/SKILL.md`

**O que mudou:** Adicionada subseção **1.2b** (detecção de STATE.md trackeado), categoria **🔧 Migrações de gitignore** no relatório da Fase 1.3 e subseção **3.7** (append no `.gitignore` sem executar `git rm --cached`).

**Como aplicar:** Copiar `skills/update-framework/SKILL.md` do framework para `.claude/skills/update-framework/SKILL.md` no projeto.

```bash
cp ${FRAMEWORK_PATH}/skills/update-framework/SKILL.md .claude/skills/update-framework/SKILL.md
```

**Impacto se nao aplicar:** Próximos `/update-framework` não vão detectar STATE.md trackeado em versões antigas — projeto continua com risco de conflito multi-dev.

### 🔧 Content patches — mudancas dentro de secoes existentes

#### 1. `CLAUDE.md` (raiz do projeto) — seção `## Regras de operacao`, regra #6

**Motivo:** STATE.md sempre foi memória pessoal de sessão; o princípio só não estava documentado. Versões anteriores deixavam ambíguo se STATE.md devia ser commitado, levando alguns projetos a tratá-lo como status de time (com conflitos).

**Texto antigo** (encontrar e substituir):
```markdown
6. **STATE.md e memoria entre sessoes.** Ao iniciar sessao → ler `.claude/specs/STATE.md`, especialmente "Em andamento" para saber a fase atual e o que falta. Ao encerrar (ou antes de `/clear`) → atualizar STATE.md com fase atual e proximos passos.
```

**Texto novo:**
```markdown
6. **STATE.md e memoria entre sessoes.** Ao iniciar sessao → ler `.claude/specs/STATE.md`, especialmente "Em andamento" para saber a fase atual e o que falta. Ao encerrar (ou antes de `/clear`) → atualizar STATE.md com fase atual e proximos passos. **STATE.md e pessoal por dev (gitignored) — cada dev mantem o seu, nunca commitar.** O mesmo vale para `.claude/plans/`: planos da fase Plan (RPI) sao pessoais ao dev que esta implementando, tambem gitignored.
```

**Para projetos em modo light** (CLAUDE.md tem `<!-- framework-mode: light -->`): a regra está numerada como **#5**, não #6. Substituir:

```markdown
5. **STATE.md e memoria entre sessoes.** Ao iniciar sessao → ler `.claude/specs/STATE.md`. Ao encerrar → atualizar STATE.md.
```

por:

```markdown
5. **STATE.md e memoria entre sessoes.** Ao iniciar sessao → ler `.claude/specs/STATE.md`. Ao encerrar → atualizar STATE.md. **STATE.md e pessoal por dev (gitignored) — cada dev mantem o seu, nunca commitar.** O mesmo vale para `.claude/plans/`.
```

**Impacto se nao aplicar:** Devs continuam sem orientação explícita sobre o caráter pessoal do STATE.md; risco de commits acidentais persiste.

### 👀 Manual (decisao humana)

#### 1. `.gitignore` (raiz do projeto)

**O que mudou no template:** Nova entrada `.claude/specs/STATE.md` no bloco de gitignore distribuído pelo framework.

**Como aplicar:** Appendar ao `.gitignore` existente:

```bash
grep -qxF ".claude/specs/STATE.md" .gitignore || echo ".claude/specs/STATE.md" >> .gitignore
```

**Atenção — se `STATE.md` já está trackeado no git** (`git ls-files .claude/specs/STATE.md` retorna a path):

Execute manualmente, **após coordenar com o time** (afeta a working tree de quem fizer pull):

```bash
git rm --cached .claude/specs/STATE.md
git add .gitignore
git commit -m "chore: untrack STATE.md (agora pessoal por dev)"
```

Após o pull, cada dev recriará seu STATE.md local (arquivo continua no working tree, só sai do tracking). O `/update-framework` deteta esse cenário e mostra os comandos, mas **não executa automaticamente**.

**Sugestao:** appendar no `.gitignore` agora (zero risco). O `git rm --cached` pode esperar até um momento de menor atividade no time.
