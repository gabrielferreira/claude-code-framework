### 📝 Structural — secoes novas/removidas

#### 1. `.claude/skills/context-fresh/README.md`

**Seção adicionada** após `### 5. Monitorar e reagir a falhas`:

```markdown
### 5b. Detecção de loop

Antes de cada retry, verificar os indicadores abaixo. **3 ou mais = loop detectado.**

| Indicador | O que checar |
|-----------|-------------|
| Mesma chamada de ferramenta com parâmetros idênticos | Tool calls repetidas sem variação de argumento |
| Mesmo erro após retry | Mensagem de erro idêntica ou equivalente na 2ª tentativa |
| 3+ abordagens distintas tentadas para o mesmo problema | Log de tentativas no completion log |
| Ciclo de arquivo (criado → deletado → criado) | Diff entre arquivos modificados nas tentativas |
| Ciclo de teste (passou → falhou → passou → falhou) | Output de testes consecutivos da mesma task |

**Quando loop detectado:**

1. **Parar** — não fazer mais retries na task
2. **Invocar `stuck-detector`** com: Task ID + lista de tentativas + erros encontrados
3. **Aguardar diagnóstico** — ler o relatório do stuck-detector
4. **Registrar no STATE.md** (seção "Blockers ativos") com root cause identificada
5. **Apresentar ao usuário** o relatório + caminhos de resolução — não avançar sem decisão humana

> **Relação com max 1 retry:** A regra "máximo 1 retry" ainda vale. A detecção de loop é uma salvaguarda anterior — para casos onde o padrão de loop é evidente antes mesmo de esgotar o retry.
```

**Impacto se não aplicar:** sem o protocolo formal de detecção de loop, o contexto-fresh pode continuar retrying indefinidamente sem escalar para diagnóstico.

### 🆕 Arquivos novos

#### 1. `.claude/agents/stuck-detector.md` (novo)

**O que é:** sub-agente de diagnóstico invocado quando a sessão principal detecta um loop de retry sem progresso. Analisa histórico de tentativas, classifica causa raiz em 5 categorias e propõe 3 caminhos de resolução. Read-only — não implementa.

**Relevante se:** projeto usa sub-agents via skill `context-fresh`.

**Como instalar:**
```bash
cp {framework_path}/agents/stuck-detector.md {project_path}/.claude/agents/stuck-detector.md
```

**Se decidir não instalar:** context-fresh continuará funcionando, mas sem diagnóstico automático de loops — falhas repetidas precisarão ser escaladas manualmente.

---

#### 2. `.claude/agents/plan-checker.md` (novo)

**O que é:** agent que valida cobertura do execution-plan contra RFs e critérios de aceitação da spec antes de iniciar implementação. Detecta lacunas entre spec→plano que causariam retrabalho.

**Relevante se:** projeto usa execution-plan + sub-agents.

**Como instalar:**
```bash
cp {framework_path}/agents/plan-checker.md {project_path}/.claude/agents/plan-checker.md
```

**Se decidir não instalar:** validação spec→plano fica manual — sem impacto se o projeto não usa plan-checker no fluxo.
