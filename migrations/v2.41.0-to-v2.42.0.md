# Migration v2.41.0 → v2.42.0

> Data: 2026-04-15
> Tipo: minor

## Resumo

Cadeia monorepo completa: setup detecta e preenche `## Monorepo` (MR2), spec-creator pergunta sub-projeto (MR3), backlog-update agrupa por sub-projeto (MR4). Scan ampliado para 2 níveis, git submodules tratados. Dual-mode: repo + Notion.

## Pre-requisitos

- Versao atual do projeto: v2.41.0
- Se estiver em versao anterior, aplique as migrations anteriores primeiro

## Mudancas

### 🔄 Overwrite (substituir arquivo inteiro)

#### 1. `.claude/skills/spec-creator/SKILL.md`

**O que mudou:** novo Passo 0d (detectar monorepo e sub-projeto), ajustes no bootstrap (Passo 0c), path condicional (Passo 2), header com Sub-projeto (Passo 3), coluna condicional no SPECS_INDEX (Passo 5). Notion mode com property Sub-projeto.

**Como aplicar:** copiar do framework.

```bash
cp {framework}/skills/spec-creator/SKILL.md .claude/skills/spec-creator/SKILL.md
```

**Impacto se nao aplicar:** `/spec` nao pergunta sub-projeto em monorepo — specs vao sempre pra raiz.

---

#### 2. `.claude/skills/backlog-update/SKILL.md`

**O que mudou:** novo Passo 0a (detectar monorepo e sub-projeto), bootstrap com path condicional, subsecoes por sub-projeto em centralizado, backlog por sub-projeto em distribuido. Notion mode com property Sub-projeto. Opcao de mover item entre sub-projetos.

**Como aplicar:** copiar do framework.

```bash
cp {framework}/skills/backlog-update/SKILL.md .claude/skills/backlog-update/SKILL.md
```

**Impacto se nao aplicar:** `/backlog-update` nao agrupa por sub-projeto em monorepo — itens ficam flat.

---

#### 3. `SPECS_INDEX.md` (template)

**O que mudou:** variante monorepo comentada (coluna "Sub-projeto" entre Owner e Fonte) adicionada como bloco de referencia.

**Como aplicar:** copiar do framework.

**Impacto se nao aplicar:** sem referencia da variante monorepo no template. Funcional — o spec-creator adiciona a coluna dinamicamente se detectar monorepo.

### 🔧 Content patches — mudancas dentro de secoes existentes

#### 4. `skills/setup-framework/SKILL.md` — Fase 3.2 (geracao do CLAUDE.md)

**Motivo:** bullet explicito para preencher `## Monorepo` na Fase 3.2 + exemplo de output. Entrada na tabela de auditoria de secoes obrigatorias.

**Relevante se:** seu projeto usa setup-framework instalado localmente (nao via plugin).

**Texto antigo** (Fase 3.2, apos bullet "Context budget"):
```markdown
- Secao "Context budget" → manter tabela por modelo (...)
- Item 8 "Validacao pre-implementacao" → manter como esta no template
```

**Texto novo:**
```markdown
- Secao "Context budget" → manter tabela por modelo (...)
- Secao "Monorepo" — **condicional** (so se confirmado monorepo na Fase 1.2):
  - Preencher `### Estrutura` com tabela de sub-projetos
  - Preencher `### Distribuicao de framework` com decisao do usuario
  - Preencher `### Convencoes de camada` com L0/L2/L3+
  - **Se single-repo:** remover a secao inteira do template
  [+ exemplo de output — ver arquivo completo]
- Item 8 "Validacao pre-implementacao" → manter como esta no template
```

**Impacto se nao aplicar:** setup pode pular `## Monorepo` ao gerar CLAUDE.md (instrucao so existia nas Fases 0/1, nao na Fase 3.2).

---

#### 5. `skills/setup-framework/SKILL.md` — Fase 0 step 5 (scan de sub-projetos)

**Motivo:** scan ampliado de 1 para 2 niveis + deteccao de `.gitmodules`.

**Texto antigo:**
```markdown
Escanear sub-diretorios (1 nivel de profundidade) procurando sinais...
```

**Texto novo:**
```markdown
Escanear sub-diretorios (ate 2 niveis de profundidade) procurando sinais...
[+ niveis de scan, exclusoes, deteccao de .gitmodules — ver arquivo completo]
```

**Impacto se nao aplicar:** sub-projetos em `apps/web/` (nivel 2) nao sao auto-detectados. Git submodules podem ser configurados acidentalmente.
